# Keepa CSV 照合モード 設計メモ

## 目的

現在の `ms_research_dev` は、卸サイトを巡回しながら Amazon / Keepa API に都度問い合わせて利益判定を行っている。

この方式はリアルタイム性がある一方で、以下の課題がある。

- Amazon 側の誤照合が起こると利益判定が崩れる
- API 呼び出しが多く、再実行コストが高い
- Keepa の CSV エクスポートを既に持っている場合、その情報を活かせていない

そこで、`Keepa CSV を親データとして取り込み、卸サイトの商品を CSV に照合するモード` を追加する。

このモードは既存の卸サイト巡回を置き換えるのではなく、`既存のリアルタイム照合モードと並存` させる。


## 結論

追加するべきモード:

- `リアルタイム照合`
  - 現行どおり Amazon / Keepa API を使う
- `Keepa CSV 照合`
  - 事前に読み込んだ CSV を使い、JAN / EAN 一致を優先して利益判定する
- 将来的に `ハイブリッド照合`
  - CSV にある商品は CSV を優先
  - CSV にない商品のみ API 照合

初期実装は、`Keepa CSV 照合` 単独モードから始めるのが安全。

さらに次段階として、

- `全サイトCSV照合`
  - 1つの Keepa CSV を読み込んだうえで
  - 複数サイトを順番に巡回し
  - 各商品の JAN を同じ CSV に当てる

を追加するのが自然。

## 基本方針

### 1. 親データは Keepa CSV

Keepa CSV を Amazon 側の基準データとして読み込む。

卸サイト側は従来どおり巡回して、

- `JAN`
- `商品名`
- `ブランド`
- `仕入価格`
- `仕入先URL`

を取得する。

その後、

- `卸JAN` と `CSV の 商品コード: EAN`

を照合し、一致した行の `ASIN / Buy Box 価格 / FBA 手数料 / 紹介料 / 月販` を使って利益判定する。

### 2. 主キーは ASIN ではなく JAN/EAN

卸サイト商品は通常 ASIN を持たないため、最初の照合キーは `JAN/EAN` にする。

実際の流れは:

1. 卸サイト商品を取得
2. 卸JAN を取り出す
3. CSV の `商品コード: EAN` と完全一致照合
4. 一致した行の `ASIN` を採用

### 3. 初期実装は完全一致のみ

最初からタイトル類似やモデル番号補助照合まで広げると誤照合が増える。

MVP では以下に限定する。

- `卸JAN == CSV EAN`

のみ採用。

一致しないものは、

- `CSV未一致`

として表示し、利益計算しない。

## 全サイトCSV照合モードの最小設計

### 目的

いまの `Keepa CSV 照合` は、

- 巡回サイトを1つ選ぶ
- そのサイトの商品JANを CSV に当てる

という流れになっている。

これを次のように広げる。

- `Keepa CSV を1回読む`
- `対象サイトを複数選ぶ`
- `各サイトを直列で低速巡回する`
- `全部同じCSVに照合する`

これにより、`今日はCSVを基準に MakeUp / ヨドバシ / カウネット / スケーターをまとめて見る` という運用ができる。


### MVP の結論

最小構成はこれ。

1. `照合モード`
   - `リアルタイム照合`
   - `Keepa CSV照合`
   - `全サイトCSV照合`

2. `対象サイト`
   - 既存の単一サイトプルダウンは残す
   - `全サイトCSV照合` の時だけ複数チェックUIを出す

3. `巡回順`
   - 並列ではなく `1サイトずつ直列`
   - サイト内も今までどおり低速

4. `結果表示`
   - 既存の一覧にそのまま流す
   - `source_site_label` で区別
   - 既存の `サイト別精度レポート` をそのまま活かす


### 採らない方針

最初は次をやらない。

- 複数サイトの同時巡回
- サイトごとの別CSV
- サイトごとの条件分岐を大きく変える
- CSV未一致時の自動リアルタイム照合

理由は、まずは `安全に・分かりやすく・放置しやすく` を優先したいから。

## バックエンド設計

### ResearchParams の拡張

今:

- `target_site: str`

追加:

- `target_sites: List[str] = []`
- `multi_site_mode: bool = False`

使い分け:

- 通常時は `target_site`
- `全サイトCSV照合` 時は `target_sites`

### 実行フロー

通常:

1. `target_site` を1つ選ぶ
2. そのサイト用 scraper を立ち上げる
3. 商品を巡回

全サイトCSV照合:

1. `target_sites` を受け取る
2. サイトごとにループ
3. 各サイトの scraper を生成
4. 各サイトの default category を巡回
5. 商品JANを同じ CSV に照合
6. 結果をまとめて保存

疑似コード:

```python
sites_to_run = params.target_sites if params.multi_site_mode else [params.target_site]

for site_key in sites_to_run:
    site_categories = get_default_categories(site_key)
    scraper = build_scraper(site_key, monitor_mode=params.monitor_mode)
    await scraper.start()
    await scraper.login()

    for current_cat in site_categories:
        async for product in scraper.scrape_products(...):
            jan = product["jan"]
            csv_match = keepa_csv_store["by_ean"].get(jan)
            ...

    await scraper.stop()
```

## 安全面

`全サイトCSV照合` にしても、以下は維持する。

- `並列巡回しない`
- `1サイトずつ直列`
- `見守りモード時も1ブラウザ系統で確認`
- `各サイトの既存の低速巡回設定を流用`

つまり、`対象サイトは増えるが、1秒間の負荷は増やさない`。

増えるのは総時間だけで、瞬間負荷はなるべく据え置く。

## 最終結論

`全サイトCSV照合` は十分可能で、しかも今の設計と相性が良い。

最小実装としては、

- `1つの Keepa CSV`
- `複数サイトをチェック選択`
- `各サイトの default category を直列巡回`
- `結果は既存一覧にまとめて保存`

がいちばん安全で実用的。

この形なら、

- 実装の差分が比較的小さい
- サイト別レポートをそのまま活かせる
- 放置運用にも向く

ので、次の実装対象としてかなり良い。

## データ構造

### Keepa CSV インデックス

CSV 読み込み時に以下の辞書を作る。

```python
{
    "4903320168589": {
        "asin": "B0FSKXMQX9",
        "ean": "4903320168589",
        "title": "...",
        "brand": "激落ちくん",
        "buy_box_price": 2430,
        "amazon_price": 2695,
        "new_price": 1710,
        "monthly_sales": 2000,
        "sales_rank": 9,
        "fba_pick_pack_fee": 425,
        "referral_fee": 374,
        "referral_fee_rate": 15.39,
        "root_category": "ホーム＆キッチン",
        "sub_category": "はたき・ほこり取り",
        "amazon_url": "...",
        "keepa_url": "...",
        "variation_asins": [...],
        "variation_attributes": "...",
    }
}
```

### 価格の優先順位

CSV 照合モードでは、Amazon 側の売価は以下順で使う。

1. `Buy Box: 現在価格`
2. `新品: 現在価格`
3. `Amazon: 現在価格`

理由:

- 通常の販売判断では Buy Box が最優先
- Buy Box が空なら新品価格
- Amazon 本体価格は補助情報

### 利益計算に使う値

- 売価: `Buy Box: 現在価格`
- 仕入価格: 卸サイト取得価格
- FBA 手数料: `FBA Pick&Pack 料金`
- 紹介料: `現在のBuy Box価格に基づく紹介料`
- ポイント: 既存の `points_rate`

初期実装では CSV に紹介料があるため、Amazon API の手数料推定は使わない。


## 利益計算ルール

CSV 照合モードの利益式:

```text
利益 = Buy Box価格 - 紹介料 - FBA Pick&Pack料金 - (仕入価格 - ポイント分)
```

既存の `calculate_roi_and_judgment()` は活かせるが、内部で紹介料を再計算しているため、以下のいずれかにする。

1. `CSV専用の利益計算関数` を用意する
2. 既存関数を拡張して `referral_fee_override` を受け取れるようにする

おすすめは `2`。

## 安全策

### 採用条件

初期実装では以下を満たしたものだけ採用する。

- `JANあり`
- `CSV EAN 完全一致`
- `売価あり`
- `FBA Pick&Pack 料金あり`

### 不採用条件

以下は `判定保留` または `CSV未一致` にする。

- JANなし
- CSVに EAN がない
- CSV行に売価がない
- CSV行に手数料がない
- 価格が 0

### 誤照合防止

初期は `タイトル曖昧一致` を入れない。

理由:

- CSV 親データ方式の価値は誤照合低減
- 曖昧一致を早く入れると、その価値を壊しやすい

## 最終判断

この CSV は `かなり使える`。

特に今回のように、

- 卸サイト価格は拾えた

という課題に対して、`Keepa CSV を親データにする方式` はかなり相性がよい。

次の実装は、`Keepa CSV 読込 + JAN/EAN完全一致の MVP` から着手するのが最善。
